Based on the **Digital Msharika (KKKT-DMP)** project blueprint, here is a clear, step‑by‑step roadmap for building the system from scratch using the **Django framework**. The plan covers backend development, database design, API creation, and integration with the proposed mobile/web frontend.

---

## 🗺️ Roadmap: Digital Msharika with Django

### Phase 0: Project Foundation
- **Set up development environment**  
  - Python 3.12+, virtual environment, Git  
  - Django 5.0+ and Django REST Framework (DRF)  
  - PostgreSQL database (as recommended in the blueprint)  
- **Initialize Django project**  
  ```bash
  django-admin startproject digital_msharika
  ```
- **Create core apps** (modular structure)  
  - `members` – handles Member Profile, Spiritual Progress, Skills  
  - `fellowships` – manages Fellowship groups and participation  
  - `pledges` – tracks financial pledges  
  - `content` – for Parish Feed, liturgy, announcements  
  - `api` – DRF views, serializers, and endpoints  

---

### Phase 1: Database Modelling (Django Models)
Translate the ERD mapping into Django models with proper relationships.

#### 1.1 Member Profile (Sections A & B)
```python
from django.db import models

class Member(models.Model):
    membership_no = models.CharField(max_length=50, unique=True)
    full_name = models.CharField(max_length=200)
    gender = models.CharField(max_length=10)
    dob = models.DateField()
    marital_status = models.CharField(max_length=20)
    marriage_type = models.CharField(max_length=50, blank=True)
    marriage_date = models.DateField(null=True, blank=True)
    marriage_place = models.CharField(max_length=200, blank=True)
    spouse_name = models.CharField(max_length=200, blank=True)
    
    # Contact
    primary_phone = models.CharField(max_length=20)
    email = models.EmailField(blank=True)
    residence_area = models.CharField(max_length=100)
    house_block_no = models.CharField(max_length=50)
    
    # Church hierarchy
    previous_parish = models.CharField(max_length=200, blank=True)
    elder_name = models.CharField(max_length=200, blank=True)
    elder_phone = models.CharField(max_length=20, blank=True)
    
    # Timestamps
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```

#### 1.2 Spiritual & Professional Progress (Sections C & D)
```python
class SpiritualProgress(models.Model):
    member = models.OneToOneField(Member, on_delete=models.CASCADE)
    is_baptized = models.BooleanField(default=False)
    baptism_date = models.DateField(null=True, blank=True)
    is_confirmed = models.BooleanField(default=False)
    confirmation_date = models.DateField(null=True, blank=True)
    takes_communion = models.BooleanField(default=False)
    
    profession = models.CharField(max_length=100, blank=True)
    education_level = models.CharField(max_length=50, blank=True)
    occupation = models.CharField(max_length=100, blank=True)
    work_place = models.CharField(max_length=200, blank=True)
    volunteer_status = models.BooleanField(default=False)  # Willingness to serve
```

#### 1.3 Fellowship & Group Participation (Section E)
Create a master `FellowshipGroup` model and a through model for memberships.

```python
class FellowshipGroup(models.Model):
    name = models.CharField(max_length=100)  # e.g., "Choir Kuu", "Uinjilisti"
    description = models.TextField(blank=True)
    leader_name = models.CharField(max_length=200)
    leader_phone = models.CharField(max_length=20)

class FellowshipParticipation(models.Model):
    member = models.ForeignKey(Member, on_delete=models.CASCADE)
    group = models.ForeignKey(FellowshipGroup, on_delete=models.CASCADE)
    is_leader = models.BooleanField(default=False)
    joined_date = models.DateField(auto_now_add=True)
    
    class Meta:
        unique_together = ('member', 'group')
```

#### 1.4 Financial Pledges (Section F)
```python
class Pledge(models.Model):
    member = models.ForeignKey(Member, on_delete=models.CASCADE)
    year = models.IntegerField(default=2026)  # Fiscal year
    building_pledge = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    stewardship_pledge = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    other_pledges = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    submission_date = models.DateField(auto_now_add=True)
    
    class Meta:
        unique_together = ('member', 'year')
```

#### 1.5 Parish Feed / Content
```python
class Announcement(models.Model):
    CATEGORY_CHOICES = [
        ('wedding', 'Wedding Bans'),
        ('funeral', 'Funerals'),
        ('general', 'General'),
    ]
    title = models.CharField(max_length=200)
    content = models.TextField()
    category = models.CharField(max_length=20, choices=CATEGORY_CHOICES)
    published_at = models.DateTimeField(auto_now_add=True)
    is_active = models.BooleanField(default=True)

class DailyLiturgy(models.Model):
    date = models.DateField(unique=True)
    readings = models.TextField()
    order_of_service = models.TextField()
    hymns = models.TextField()
    verse_of_the_day = models.CharField(max_length=300)
```

### Phase 2: Admin Interface & Data Migration
- **Register all models** with Django admin for easy data entry by church staff.  
- Customise admin forms to mimic the original paper forms (fieldsets, inlines).  
- Create **data import scripts** to migrate existing paper records (CSV/Excel upload).  
- Set up user authentication:  
  - Admin users (church staff) – can manage all data.  
  - Members – will later have limited access via the mobile app.

---

### Phase 3: REST API with Django REST Framework
Build the API that will power the mobile frontend (React Native) and any future web dashboard.

#### 3.1 Core Endpoints
- `/api/members/` – list and detail (with nested spiritual progress, fellowships, pledges)  
- `/api/fellowships/` – list groups and members  
- `/api/pledges/` – submit/view pledges  
- `/api/announcements/` – filter by category  
- `/api/liturgy/today/` – today’s liturgy and verse  
- `/api/skill-search/?profession=<value>` – search members by profession/occupation  

#### 3.2 Authentication & Permissions
- Use **TokenAuthentication** or **JWT** for mobile users.  
- Members can only view/update their own profile.  
- Staff have full access.

#### 3.3 Additional Features
- **QR code generation** for digital membership card (include member ID and photo).  
- **Push notifications** – integrate with Firebase Cloud Messaging (FCM) for announcements and verse of the day.  
- **Localization** – use Django’s internationalisation (`i18n`) to serve content in Swahili, English, and German.  
- **Multi‑tenancy** – if other dioceses/denominations will use the system, add a `Tenant` model and isolate data.

---

### Phase 4: Mobile App (React Native) – Optional but Recommended
While Django handles the backend, the blueprint’s frontend is React Native. You can either:
- Develop the mobile app separately (using the API).  
- Or start with a responsive Django web interface (Bootstrap/HTMX) for quick MVP.

#### 4.1 Core Mobile Features
- **Member Dashboard** – display spiritual roadmap progress bars, digital ID with QR code.  
- **Fellowship Finder** – use member’s residence area to suggest nearby groups, one‑tap call/chat with elder.  
- **Parish Feed** – list announcements, daily liturgy, verse of the day.  
- **Skill‑Based Marketplace** – searchable directory of members by profession.

---

### Phase 5: Testing & Deployment
- **Write unit tests** for models and API endpoints.  
- **Set up CI/CD** (GitHub Actions) for automated testing.  
- **Deploy**  
  - Backend: Use a production server (e.g., DigitalOcean, AWS EC2) with Gunicorn + Nginx.  
  - Database: PostgreSQL (managed service like RDS or self‑hosted).  
  - Media storage: AWS S3 or similar for member photos.  
- **Environment variables** for secrets (DB credentials, API keys).  
- **SSL certificate** (Let’s Encrypt) for secure API.

---

### Phase 6: Rollout & Training
- Pilot with one parish to validate data migration and user experience.  
- Train church administrators on using Django admin and data entry.  
- Launch mobile app to members via Google Play Store (and Apple App Store if needed).  
- Gather feedback and iterate.

## 📌 Summary of Django‑Specific Advantages
- **Batteries‑included admin** – perfect for church staff who need to manage data without coding.  
- **ORM** – clean mapping from the blueprint’s tables to Python models.  
- **DRF** – rapid API development for the mobile frontend.  
- **Built‑in i18n** – easy Swahili/English/German translations.  
- **Scalable** – can later add more parishes as tenants with Django’s multi‑db or schema support.
